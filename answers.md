1. Multi-stage builds in a Dockerfile are used to significantly reduce the final image size and improve security. This approach separates the build stage from the production stage, ensuring that the final image contains only the runtime artifacts and excludes unnecessary build tools. By removing these tools—which may introduce vulnerabilities or insecure utilities—the attack surface is reduced, lowering the risk of container exploitation and potential impact on the underlying host system.

2. When a developer pushes code to a version control system (eg. GitHub, GitLab, BitBucket), the pipeline is triggered automatically. In the Continuous Integration (CI) stage, the source code is checked out (lint with ruff/ black), dependencies are installed, the application is built, and automated tests such as unit and integration tests are executed. Static code analysis and security checks may also be performed to ensure code quality (with sonarqube). If any step fails, the pipeline stops and feedback is provided to the developer.

Once the code passes all CI checks, it is packaged into a deployable artifact, such as a binary or container image, and stored in an artifact or container registry (Docker Hub, GitHub registry, AWS ECS). In the Continuous Delivery (CD) stage, the artifact is deployed to a non-production environment (e.g., staging or UAT) where further validation such as end-to-end, performance, or security testing is performed. After successful validation (and optional manual approval), the application is deployed to the production environment using a safe deployment strategy such as rolling, blue-green, or canary deployment. Finally, post-deployment health checks and monitoring are used to verify system stability, with rollback mechanisms in place if issues are detected.
