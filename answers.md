1. Multi-stage builds in a Dockerfile are used to significantly reduce the final image size and improve security. This approach separates the build stage from the production stage, ensuring that the final image contains only the runtime artifacts and excludes unnecessary build tools. By removing these tools—which may introduce vulnerabilities or insecure utilities—the attack surface is reduced, lowering the risk of container exploitation and potential impact on the underlying host system.

2. When a developer pushes code to a version control system (e.g., GitHub, GitLab, Bitbucket), the pipeline is triggered automatically. In the Continuous Integration (CI) stage, the source code is checked out, dependencies are installed (with pip, Poetry, npm, etc.), and the code is linted and formatted (with Ruff, Black, etc.). The application is then built and automated tests such as unit and integration tests are executed (with pytest, JUnit, or Jest). Static code analysis and security checks are also performed to ensure code quality and catch vulnerabilities early (with SonarQube for quality, Trivy for dependency scanning). If any step fails, the pipeline stops and feedback is provided.
Once the code passes all CI checks, it is packaged into an immutable deployable artifact, such as a binary or container image, and stored in an artifact or container registry (Docker Hub, GitHub Container Registry, AWS ECR). In the Continuous Delivery (CD) stage, the same artifact is deployed to a non-production environment, such as staging, where further validation such as end-to-end, performance, or security testing is performed. After successful validation and optional manual approval, the application is deployed to the production environment using a safe deployment strategy such as rolling, blue-green, or canary deployment, often combined with feature flags. Finally, post-deployment health checks and monitoring are used to verify system stability, with rollback mechanisms in place if issues are detected.

3. The quality gate is enforced through a two-step sequence inside the sonarqube job. In the first step, the sonarqube-scan-action@v3 runs the scanner, uploads the source code, coverage data, and test results to the SonarQube server (which is self-hosted in this scenario). Because sonar.qualitygate.wait=true is set in sonar-project.properties  polls the server and blocks until the gate result is returned. In the second step, the sonarqube-quality-gate-action@v1.1.0 reads .scannerwork/report-task.txt (written by step 1) and surfaces an explicit FAILED status directly in the GitHub Actions UI.
The gate checks coverage from one specific source rather than the whole test matrix. Even though tests run across multiple Python versions and operating systems, only the run on Python 3.12 (primary version) / ubuntu-latest uploads its coverage-report artifact, which the sonarqube job downloads and sends to the server.
If the gate fails, everything after it stops. The jobs are wired together with needs: so they run in order (lint -> test -> sonarqube -> build -> deploy), and a failure at sonarqube breaks the chain. The job exits with an error, the build and deploy jobs are skipped entirely and the pull request shows a failed check that blocks merging.
